## LLM

LLM（Large Language Model，大语言模型，简称大模型）指通过海量文本预训练得到的通用语言模型，使用上可以理解为一个基于自然语言理解的处理输入文本序列后生成输出文本序列的工具。

### Token

Token（词元）是LLM处理文本序列的基本单元，类似于人类处理自然语言时理解的字词，但两者并不等同，Token的具体拆分方式依赖于Tokenizer内置的分词规则与词汇表。Token划分的精细度会直接影响模型的理解能力：特别是在中文处理时，分词歧义（同一字符序列的多种切分方式）和生僻字/低频专业术语的切分粒度，会直接影响模型的语义理解效果。

Tokenizer（分词器）是LLM的"翻译官"，它的核心作用是对人类可读的自然语言文本做预处理，将其转换为LLM可理解的数字序列。为了在"词表大小"和"序列长度"之间取平衡，分词采用子词切分算法：它把高频词保留为整体，把低频词拆成更小的片段。

压缩比指**输入序列字符数**与**分词后Token数**的比值（可理解为平均每个Token覆盖多少字符）。压缩比越大，越有利于降低模型推理成本和提升模型推理速度，但过度追求压缩比会导致词汇表过大，其中生僻词在训练数据中的出现频率过低，导致模型泛化能力差。

### 自回归生成

**自回归生成**是LLM的核心运行机制：在调用LLM API时，用户输入会先经**Tokenizer**处理拆分为Token序列并映射为Token ID序列（一组整数序列）作为LLM输入，LLM会根据输入Token ID序列预测下一个最可能出现的Token ID，生成的Token ID会被追加到序列末尾，如此循环往复，最后输出的Token ID序列会被**Tokenizer**解码为自然语言文本序列，作为输出返回给用户。

### 多模态

多模态指LLM支持处理非文本的能力。在输入LLM前，Tokenizer会通过占位符标记多模态（图像/音视频）输入，它们会被编码器转换为特征向量后经过投影层映射为软Token ID序列，然后经Tokenizer编码形成的文本Token ID序列和这些软Token ID序列会被拼接后输入LLM；LLM通过注意力机制理解图文关系，并预测生成Token。这里主要说的是输入侧多模态，目前已经相当普及，而输出侧多模态由于计算量大、一致性差、难以评估等障碍仍在发展中。多模态输入/输出会占用大量Token。

### 上下文窗口

上下文窗口是单次调用里输入与输出（合计或分计，依厂商说明）可处理的Token数上限，即模型的"工作记忆"。上下文窗口的大小决定了模型的对话连续性与单次处理能力。上下文窗口往往存在隐性占用，因为它不只包含User Prompt（用户输入），还包含System Prompt（调节模型行为的系统指令）、对话历史（过往的消息记录）、RAG检索结果（从外部知识库检索到的补充信息）、工具调用模式定义（Schema，函数定义与参数结构）、格式开销（特殊字符、换行符、Markdown标记等）和模型生成的输出Token（详见自回归生成机制）。

上下文窗口受限于Transformer架构自注意力机制，输入序列过长会导致计算成本过度增长（计算时间复杂度为O(n^2)）、推理延迟增加、安全风险增加等问题。上下文接近上限时，会出现模型忽略早期约束、"中间丢失"现象、回答漂移、RAG失效、成本与延迟激增等问题；避免此问题的常见手段是智能截断和分批处理 + 二次汇总。

### 解码与采样参数

自回归生成中，模型每一步会对词表中每个候选Token打分(logits)；logits经softmax数学变换为概率分布，再按该分布采样决定下一个Token。采样参数用于在这个过程中施加控制。

#### Temperature

Temperature用于控制模型的"冒险程度"，它越高则模型输出随机性越高：原理为在softmax前将logits除以它（对于Temperature为0的场景不能直接除，主流存在两种解决方案：统一给Temperature加一个极小值；在0值时直接走边界处理逻辑），从而改变采样时候选Token的概率分布。

#### Top-k与Top-p

Top-k和Top-p用于缩小"抽签池"，它们会直接砍掉不靠谱的候选项：Top-k固定只保留概率最高的k个候选，Top-p从高到低累加概率，保留累计达到阈值p的最小候选集合再采样。

#### Penalty

Penalty系列对已经出现过的词降分以防止"复读机"：Repetition Penalty下调当前序列里已出现过Token的logits，且出现次数越多的Token下调越多，惩罚力度是指数增长的；Presence Penalty对所有出现过的Token施加惩罚，但与出现次数无关；Frequency Penalty按Token出现次数加重惩罚，惩罚力度是线性增长的。

### 思维链模式

思维链通过引导模型逐步推理提高复杂题正确率，部分模型支持"思维链模式"，此时采样参数通常会被忽略，这是因为思维链模式旨在让模型"自由思考"，所以只会采用事先固定的采样策略。所以调用思维链模型时，应当通过提示词而非采样参数来约束模型输出。

### Streaming

Streaming（流式输出）是在模型每生成一个或几个Token时就会立即推送给客户端，即边生成边返回，主要是通过降低**首字延迟（TTFT）**来改善用户体验，但它不会降低总耗时和计费成本，目前已经普及成为行业标准和默认配置。

#### 流式协议

流式输出的实现需要依赖传输协议，常见的协议如下：
1. SSE（Server-Sent Events，服务端事件推送）：浏览器原生支持的服务端到客户端单向数据流推送协议，使用`text/event-stream`作为Content-Type，以UTF-8纯文本格式传输。常用字段包括`data`（业务载荷，允许多行）、`event`（自定义事件名，浏览器默认为message）、`id`（事件序号，配合浏览器重连语义做断点续传提示）和`retry`（建议的重连间隔毫秒数）。核心优势是简单，浏览器端几行代码即可接收事件，适合文本聊天、模型增量输出、状态通知等场景；但由于采用连续两个换行符`\n\n`作为事件分隔符，正文片段含有未转义换行符`\n`会导致客户端解析错误。
2. WebSocket：全双工长连接协议，客户端和服务端可以随时互发消息。适合用于高实时性要求的复杂交互，如实时语音、多人协作、需要频繁取消或插话等场景；但连接管理复杂，需要自行处理网关、鉴权、心跳等。
3. HTTP Chunked（分块传输编码）：HTTP/1.1的传输机制，响应体在没有`Content-Length`的情况下分块发送，能实现"边写边发"。适合后端到后端流式代理、低层传输等；但由于它偏底层，本身不定义应用层的事件类型、重连语义和消息边界，需要业务层自行设计协议。目前是用于承载SSE的主流传输机制。
4. HTTP/2流式输出：基于HTTP/2协议的多路复用和二进制帧（Frame）拆分机制实现的流式传输。优点是低延迟，支持高并发和双向数据流。部分大模型API供应商已支持基于HTTP/2的gRPC流式接口，劣势在于浏览器对原始HTTP/2帧的直接操作支持有限。可以采用HTTP/2作为SSE的传输机制。

#### 流式异常

流式链路最容易出问题的地方，往往不是“怎么开始”，而是“怎么结束”。常见的流式异常有以下四类场景：
1. 用户取消：用户关闭页面、点击停止生成、切换会话，都应该触发取消；后端需要同时取消到供应商API的请求、正在解析的响应流、后续TTS/工具调用/落库任务以及未提交的增量缓存；只停止前端展示而不停止后端操作，会导致无谓的Token成本浪费。
2. 超时：超时至少分为连接超时（连不上供应商）、TTFT超时（连接建立但迟迟没有第一个事件返回，通常指向模型排队、上下文过长或供应商抖动）、总时长超时（一直在输出但超过业务可接受时间，可能只是用户让模型写太长）三层；三者要分开记录和处理，因为根因和处置策略不同。
3. 断流：流式响应在没有正常结束的情况下中断，此时不要轻易把半截内容当作成功。正确做法是记录`finish_reason`或最后事件状态，若无正常结束标记则将本次调用标记为INTERRUPTED，前端展示"已中断，可重新生成"，不要悄悄存成完整答案。
4. 重连：SSE的EventSource有自动重连能力，但大模型流式输出在重连后能否从断点续传取决于服务端是否保存了事件序号、增量片段和供应商调用状态。多数情况下，供应商侧流断开就无法真正从Token级别续传。稳妥起见，可以采用以下做法：
   1. 服务端为每个流式响应生成`messageId`和递增`sequence`。
   2. 将已发送片段写入短期缓存。
   3. 前端重连时先补发已缓存片段。
   4. 若供应商流已结束或失效，提示用户重新生成，不要假装无缝续写。

### Logprobs

Logprobs是每个生成Token的对数概率，可以理解为模型对Token的"确信程度"，它主要用于置信度评估、异常检测、多候选对比。Logprobs只有部分供应商的API支持，且会增加响应体积。

## Agent

**AI Agent**（人工智能智能体）指以LLM为决策核心、能在开放任务中结合工具与记忆、通过多轮循环持续感知环境并调整行动的自主软件系统。传统编程和工作流都是人在做决策、提前设计好所有逻辑，而Agent是AI在做决策，这是Agent区别于前两者的本质区别，其他差异也都是源自这一点。与传统编程和工作流解决线性逻辑问题（采用预设路径解决确定性问题）不同，Agent解决的是非线性逻辑问题。不同于单纯的聊天机器人，AI Agent强调自主性和交互性，能够在动态环境中持续迭代，直到任务完成或终止。

核心公式：Agent = LLM + Planning（规划） + Memory（记忆） + Tools（工具）

Agent Loop是所有Agent范式共享的运行引擎，本质是一个循环，每次迭代完成LLM推理->工具调用->上下文更新，直到任务终止。为防止模型陷入死循环，需要设置强制中断条件，如最大迭代轮次上限或Token消耗阈值。

构建Agent系统的工程框架通常由以下三大部分组成：

1. LLM Call（模型调用）：底层API管理，负责抹平各大厂商LLM的接口差异、处理流式输出、Token截断、重试机制等基础能力。
2. Tools Call（工具调用）：解决LLM如何与外部世界交互的问题，主要包括Function Calling、MCP（Model Context Protocol）和Skills这3种机制。
3. Context Engineering（上下文工程）：管理传递给LLM的Prompt集合。狭义上指系统提示词的编排，广义上还包含动态记忆注入、用户会话状态管理、工具与Skills描述的动态组装等。

其中Context Engineering最易被忽视但价值最高，也是目前Agent开发的核心难点。

### 核心范式

#### ReAct模式

**ReAct模式**是当前AI Agent理论中最具基础性和代表性的范式，它将"思维链推理"与"外部环境交互行为"结合，弥补单纯LLM缺乏实时信息和容易产生幻觉的缺陷。通过交织推理和行动，ReAct使模型生成更可靠、可追踪的任务解决轨迹，提高解释性和准确性。它让AI在整体目标的指引下"走一步看一步"，打破了一次性规划全部流程的局限，通过动态的交替循环边思考边验证。

#### Plan-and-Execute模式

**Plan-and-Execute模式**让LLM充当规划者，先制定全局的分步计划，再由执行器按步骤逐一完成，而非"边想边做"。它非常适合步骤繁多、逻辑依赖明确的长期复杂任务，能有效避免ReAct模式在长任务中容易出现的"迷失"或"死循环"问题。可以与ReAct模式结合使用。

#### Reflection模式

**Reflection模式**赋予Agent自我纠错与迭代优化能力，在产出或行动后通过自然语言形式的口头反馈强化模型行为，显著提升了Agent在不确定环境下的鲁棒性，但会带来额外的LLM调用开销。它通常不单独使用，而是作为增强层叠加在ReAct或Plan-and-Execute之上。ReAct + Reflection使每轮观察后不仅更新行动计划，还进行显式自我反思，形成自适应Agent。

#### Multi-Agent系统

**Multi-Agent**（多智能体）系统是指多个独立Agent通过协作完成单一复杂任务的架构，每个Agent专注于特定角色或职能，类比人类的团队分工协作。核心架构模式分为中心化的Orchestrator-Subagent模式（编曲者模式）和去中心化的Peer-to-Peer模式（编舞者模式）。优势在于通过并行处理提升复杂任务效率、通过专业化分工提升各模块准确率、单个Agent失败不影响整体架构的健壮性、易于新增专项Agent的高度可扩展性，缺点在于Agent间通信开销高、协调失败可能导致任务全局崩溃、调试与监控难度大、多LLM调用导致成本上升等。在Multi-Agent中，Agent之间如何高效、低成本通信是一个工程难题，A2A（Agent-to-Agent）协议正是为解决此问题而生。A2A协议是专门为AI Agent间高效、确定性协作而设计的通信规范，它要求Agent使用高度结构化、带有严格校验规则的数据载体通信，相当于给大模型之间定义了接口契约。

#### Agentic Workflow

**Agentic Workflow**（智能体工作流）是对上述所有范式的终极整合，核心思想是围绕LLM设计一套工作流，涵盖四大核心设计模式：
1. Reflection（反思）：让模型检查自己的工作。
2. Tool Use（工具使用）：为模型配备与外部世界交互的工具（ReAct中的Acting）。
3. Planning（规划）：让模型提出多步计划并执行（即Plan-and-Execute）。
4. Multi-Agent Collaboration（多Agent协作）：多个不同的Agent共同工作。

核心在于用软件工程思维将这些模式组合编排，从而构建强大的AI应用。

### Tools

Tools是Agent提供与外部世界交互的工具，主要涉及三种抽象机制：**Function Calling**解决「模型如何把意图变成一次结构化调用」；**MCP**解决「工具与数据如何跨宿主、跨模型标准化接入」；**Skills**则把领域里的流程与规范沉淀成可发现、可按需加载的上下文单元，且常与前二者组合使用。

#### Function Calling

**Function Calling**（函数调用）是LLM输出结构化调用的能力，本质上就是LLM识别和使用特定数据结构的能力。模型没有直接执行你的后端代码，它实际上是根据用户问题和工具描述生成一个结构化的工具调用意图，工具的实际调用由应用侧完成。换句话说，由于LLM内部并不能真的运行函数，实际上这是将函数的使用说明注入上下文让其理解如何使用函数；模型会根据用户请求和函数使用说明决定是否调用函数，如果确定调用则返回结构化调用以指定调用哪个函数并提供参数，具体的函数调用是由外部执行的。Function Calling的价值是让模型完成从自然语言意图到结构化参数的映射，但它只负责映射，不负责执行具体的操作，概括来说就是把模型擅长的语义理解和程序擅长的确定性执行连接起来。当前，业内普遍采用OpenAI Function Calling Schema作为统一规范处理Tools描述，通过JSON Schema严格定义工具的描述和参数规范，使LLM在推理时理解工具的功能边界，从而决定"是否调用"以及"如何填充参数"。从工程视角看，一次完整的Function Calling链路包含七个步骤：服务端注册工具定义、用户发起请求、模型选择工具、业务侧校验参数、业务侧执行工具、工具结果回填模型、模型生成最终回答。

#### MCP

**MCP**（Model Context Protocol，模型上下文协议）是外部系统接入LLM的标准，它用JSON-RPC 2.0统一了LLM与外部数据源/工具的通信规范，解决了AI应用开发中的复杂性和碎片化问题，核心价值在于解耦和标准化。MCP v1.0定义了四种核心能力类型，以覆盖LLM与外部交互的主要场景：
1. **Resources（资源）**：定义只读数据流，让模型能像读取本地文件一样读取外部数据
2. **Tools（工具）**：定义可执行动作，用于声明模型可以主动触发的代码或API
3. **Prompts（提示模板）**：定义预设指令集，是MCP服务器提供给模型的"标准化操作指南"
4. **Sampling（采样）**：让MCP Server能反向请求Host侧的LLM进行推理生成，打破了单向数据流，使其能够按需调用LLM智能处理数据。

MCP采用JSON-RPC 2.0作为应用层通信协议，具有轻量易接入、支持多种传输协议、纯文本格式便于人工阅读调试、被多种编程语言广泛支持的优点；它采用面向操作的语义模型和严格的JSON数据格式，天生适合用于系统内部通信和工具调用场景。MCP支持多种传输方式，通常为以下三种：
1. stdio（标准输入/输出）：适用于本地进程间通信，优势在于极度轻量、无网络开销、启动快，通常用于桌面应用集成和本地工具。
2. Streamable HTTP（流式HTTP）：适用于远程部署的独立服务间通信，优势在于标准兼容性好、每条请求独立鉴权、无需维护长连接，通常用于WEB应用、团队共享MCP服务、云端托管MCP服务等。
3. SSE（Server-Sent Events，服务端事件推送）：适用于单纯服务端推送通知而不需要与客户端双向通信的场景，由于需要维护持久连接、处理鉴权复杂（除建立连接外，后续消息无法使用Authorization等请求头字段）、与负载均衡器/API网关兼容性差等问题，建议在新项目中改用流式HTTP。

#### Skill

当多个原子工具需要在特定场景下被反复组合调用时，可以将这一调用序列封装为一个Skill（技能），对外暴露为单一的可调用接口。在工程落地中，Skill演化出了两种核心形态：
1. Toolkits（黑盒形态）：将多个原子工具在代码层封装为高阶工具，对外暴露单一的JSON Schema。LLM只能看到函数签名和参数描述，无法感知内部实现逻辑。核心价值是降低推理步骤和Token消耗，适用于逻辑固定、调用路径明确的场景。
2. Skills（白盒形态，主流趋势）：指用自然语言定义的、具有特定领域上下文的逻辑指令集，它通过延迟加载优化Token消耗以缓解上下文窗口压力。其核心价值是把隐性规则变成显性的文档，固化为可复用的SOP（Standard Operating Procedure，标准作业程序），用于指导Agent处理复杂灵活的任务。Skill的关键机制为**延迟加载**与**动态上下文注入**，它用自然语言将决策权下放给模型，模型在动态加载元数据（通常用SKILL.md声明）后，将其中定义的规则、流程和约束实时注入到推理上下文中，指导后续的工具调用和决策。Skills没有创造新能力，但通过一套提示词规范定义了AI能力模块化复用的标准，解决了复杂任务如何编排的问题。编写高质量Skills需要遵循五大核心原则：
    1. 语义精确的元数据：要求消除歧义、明确边界、并融入意图触发词；操作上可以从“描述功能”转向“定义场景、问题和触发条件”。
    2. 模块化与单一职责：按排查维度拆分，以确保每个Skill单一职责；操作上要避免单一“系统故障排查器“，改为工具集。
    3. 确定性优先原则：LLM负责提取参数，脚本负责逻辑闭环；对于需要严谨逻辑的计算或格式转化，永远不要相信LLM的“直觉”，要让它去驱动脚本。
    4. 渐进式披露策略：避免“信息过载”导致Agent迷失；通过文档的分层结构，让Agent只在需要时加载细节。
    5. 极简主义：避免浪费上下文窗口容量，提高信噪比。

### Memory

**Memory**指Agent的记忆系统，为Agent提供了"跨会话的状态保持"和"基于经验的持续进化"能力。

记忆系统是一个动态演化的有机体，其生命周期包含以下核心操作：
1. 编码：将原始交互转化为可存储的结构化信息。
2. 存储：将编码后的信息持久化。
3. 提取：根据上下文检索相关记忆。
4. 巩固：将短期记忆转化为长期记忆。
5. 反思：主动回顾评估记忆内容，优化决策。
6. 遗忘：淘汰低价值或过时记忆。

最新定义的记忆系统包含控制策略、时间跨度、表征形式三大维度，核心问题是"何时写"、"何时读"、"何时更新"：传统记忆系统采用规则触发（如每轮对话后写入）；前沿方案则尝试通过强化学习让Agent学习最优的控制策略，以判断当前信息是否值得写入、何时触发记忆更新、何时主动遗忘等，这使得Agent能够根据任务特点自主调整记忆行为，避免依赖硬编码规则。

#### 记忆分类

从时间跨度看，记忆系统可以分为**短期记忆（Session 级）**与**长期记忆（跨 Session）**，前者让Agent在单次对话中保持连贯性，后者让Agent能跨越多个会话（Session）积累用户偏好和历史经验。

从存储形式看，记忆系统可以按存储位置与表征形式分为Token级记忆（以自然语言或离散符号形式存储在外部数据库）、参数化记忆（将信息编码进模型参数中）和潜在记忆（以隐式形式承载在模型内部表示中，临时存在于内存/显存中），且这三种形式间可以相互转换。

按功能目的，记忆分为事实记忆（Agent知道什么，包括用户偏好、环境状态、显式事实）、经验记忆（Agent如何改进，包括过往轨迹、成败教训、策略知识）、工作记忆（Agent当前思考什么，包括当前推理上下文、任务进展）。按内容性质，记忆分为情景记忆（发生了什么，记录特定时间、场景下的具体事件）、语义记忆（意味着什么，从多个情景中提炼的通用知识、事实或规律）和程序记忆（存储技能、规则和习得行为，使Agent能自动执行任务序列而无需每次显式推理）。

#### 短期记忆

短期记忆是Agent在当前单次会话（Session）中持有的暂存信息，涵盖用户的提问、模型的每轮回复，以及工具调用的中间结果（Observations），它直接作为Prompt的组成部分参与LLM当前轮次的推理，是 Agent感知"当前任务上下文"的唯一来源。短期记忆依托LLM自身的上下文窗口实现，但更大的上下文窗口并不意味着可以无限堆砌上下文，为了缓解推理成本增长与"中间丢失"现象，需要在上下文工程中主动控制输入信息的分布。为控制短期记忆的膨胀，框架层通常在运行时采用以下三类压缩策略：
1. 上下文缩减（Context Reduction）：当对话历史达到预设Token阈值时，框架自动丢弃最早的N轮消息（滑动窗口算法）或调用轻量模型将历史对话总结压缩，以最小的信息损耗换取上下文空间。
2. 上下文卸载（Context Offloading）：将工具或Skill调用返回的大体量数据挂载到外部临时存储，Prompt中只保留极短的引用标识；当模型需深挖细节时，通过强制关联的Function Calling触发内部工具执行读取动作。
3. 上下文隔离（Context Isolation）：在多Agent架构中，主Agent向子Agent分配子任务只传递精简的任务指令和必要的上下文片段，避免将整个对话历史广播给每个子Agent，这是控制多Agent系统总Token消耗的关键工程实践。

#### 长期记忆

长期记忆是跨越单个会话（Session）的持久化知识与经验库，它不随对话结束而销毁，通过主动的"写入-检索"机制使Agent能在全新的会话（Session）中调取历史沉淀的用户偏好、事实认知和过往经验。长期记忆的实现原理是记忆写入（Record）与记忆检索（Retrieve）的双向交互：
1. 记忆写入（Record）：对话结束后，框架触发后台异步任务，调用LLM对本轮短期记忆进行语义"提纯"（过滤冗余的对话噪声，抽取高价值的结构化事实），以结构化条目的形式写入持久化存储。
2. 记忆检索（Retrieve）：在新会话（Session）开始时，用户查询（Query）被向量化，与长期记忆库中的条目进行语义相似性检索，将最相关的历史偏好和背景知识注入到当前会话（Session）的系统提示（System Prompt）中，使Agent在对话伊始就具备"了解这位用户"的上下文感知能力。

在基础的写入与检索之上，生产级Agent系统还需要一套演化机制，来保证记忆的质量与检索的信噪比。
为了实现将"原始数据"转化为"高阶知识"，Agent不仅被动记录原始对话，还需要定期对记忆进行自我审计和二次加工，实现方式主要包括：
1. 自我反思（Self-Reflection）：在任务完成后，Agent启动一个异步任务，复盘本次任务的成败原因，并将"教训"提取为一条 元知识（Meta-Knowledge）。
2. 精细化反思闭环（Reflect Loop）：将反思机制演化为更精细的"规划-执行-反思-记忆"闭环。反思不仅发生在任务完成后，而是在每个子任务结束时都会触发。独立的Reflect Agent会对子任务输出进行真实性验证（输出是否符合客观事实）、交付物验证（是否完成了用户指定的目标）和数据保真性验证（关键数据是否在传递过程中丢失或变形）这三重验证。通过这种细粒度的反思，可以有效防止错误在多轮推理中累积放大。
3. 记忆聚类与合并（Clustering & Consolidation）：当长期记忆中出现大量碎片化、重复的记录时，系统会自动触发合并任务，将这些碎片合成为一个完整的"实体百科"，减少存储冗余并提升检索一致性。

为了避免"记忆爆炸"和"过时记忆干扰"，会采用以下工程策略：
1. 权重与衰减（Importance & Decay）：为每条记忆维护一个合并了近期性（Recency）、重要性（Importance）、相关性（Relevance）的综合权重得分，它在从向量库完成静态语义召回后会作为重排依据用于调整候选项得分。
2. 主动冲突解决：当新旧记忆发生逻辑冲突时，系统需识别并标记旧记忆为"废弃状态"，防止Agent给出过时的建议。

由于长期记忆系统涉及向量化存储、语义检索和记忆管理等复杂逻辑，通常将其剥离为独立的第三方组件，其底层架构通常由以下三层组成：
1. VectorStore（向量数据库）：将提取的记忆文本转化为语义向量（Embeddings）存储，常见方案包括Pinecone、Weaviate、Chroma、Qdrant等。
2. GraphStore（图数据库）：将记忆以"实体-关系"的形式建模为知识图谱（如采用Neo4j存储），适用于需要多跳推理的复杂查询场景。
3. Reranker（重排序器）：向量检索的初步召回结果在语义相关性上并不精确有序，Reranker基于交叉编码器（Cross-Encoder）对召回结果进行二次精排，显著提升最终注入上下文的记忆质量。

## Prompt Engineering

**Prompt Engineering（提示词工程）**是通过系统化地设计和迭代输入指令，优化大模型输出质量的工程方法论。成功的Prompt都是经过初始版本→测试→调优→再测试的循环打磨出来的。

### Prompt的本质与作用

Prompt（提示词）的本质是给大语言模型下达的指令，作用是引导模型走向正确的Token序列。Prompt并不是越详细越好，过于冗长的Prompt会产生稀释焦点（模型需要在大量无关信息中找到真正重要的指令）、增加幻觉风险（指令越多，模型越容易"自以为是"地补充细节）、拖慢推理速度（更长的 context 意味着更高的延迟和成本）等问题，所以编写Prompt的核心原则是用最简洁的语言精准传递意图。

### Prompt的核心四要素框架

一个合格的Prompt通常包含以下四个核心要素（即采用四要素框架）：
1. **Role（角色）**：激活模型的相关知识领域。
2. **Task（任务）**：明确要完成的具体动作。
3. **Context（上下文）**：提供任务相关的背景信息。
4. **Format（格式）**：指定输出的结构要求。

由于大模型的"中间丢失"效应，将角色定义放在开头、格式要求放在结尾可以有效利用大模型更容易关注上下文开头和结尾信息的特点。

### Prompt的六大核心技巧

#### 角色扮演（Role-Playing）

给模型一个明确的专家身份，能让回答更专业、更有针对性。原理是大模型的训练数据中，不同领域的内容有不同的分布特征，通过设置角色可以使模型激活相关的知识子空间，进而使输出的内容会更精准、更符合该领域的表达习惯。角色定义的粒度越精准则效果越好，另外需要注意"角色疲劳"问题：如果在一个长对话中反复使用同一个角色，模型的"角色感"会逐渐减弱；建议对复杂任务使用专门的新对话，让角色激活更纯粹。

#### 思维链（Chain-of-Thought, CoT）

CoT是处理所有需要推理的复杂任务时的核心技巧。它通过强制逻辑推导（模型在输出最终答案前，需要完成更充分的中间推理步骤）、过程透明（推理步骤可见，便于调试Prompt或验证结论可靠性）、对抗幻觉（展示推导过程以提高编造事实的成本）来保证有效性。CoT有三种形态，包括Zero-shot CoT（基础CoT，简单任务效果不错）、引导式CoT（推荐）和结构化CoT（最强，基于XML标签结构编写）。在复杂推理任务上，使用CoT往往比直接给出答案的准确率更高。

#### 少样本学习（Few-Shot Learning）

对于复杂或格式严格的任务，提供1～3个示例比纯文字描述更有效。原理是示例相当于隐性格式规范，让模型从示例中能学到"输出应该长什么样"。示例选择的原则是保证相关性（示例必须与实际任务属于同一类型）、多样性（覆盖主要的边缘情况和潜在挑战）和清晰性（使用XML标签包装示例，保持结构）。示例数量上：1个示例适用于简单明确的格式要求；2～3个示例适用于复杂格式或多种边缘情况；超过3个则收益递减，徒增Token成本。

#### 任务分解（Task Decomposition）

对于极其复杂的任务，将其分解成更小、更简单的子任务，让模型逐一完成后再汇总，本质上是采用算法中的分治思想解决上下文窗口上限问题和模型注意力不足问题。任务分解可以分为静态分解（任务开始前完整规划子任务序列，适用于流程固定的场景）和动态分解（执行过程中根据输出动态决定下一步，适用于探索性、分析性任务）两种类型。调试的核心技巧在于：如果模型在某一步总出错，将该步骤单独拎出来调优，而不是重写整个任务链。

#### 结构化输出（Structured Output）

要求模型以特定格式输出，并在Prompt中明确给出Schema。通常可以选择使用JSON（可直接序列化传输；但语法严格，解析失败需重试）、XML（层级清晰，可读性好；但体积较大）、YAML（流式友好，体积小；但对缩进敏感）、Markdown（可读性好，适合展示；但解析复杂）等格式；另外，现代模型越来越多地原生支持结构化输出，此时JSON Schema直接发送给模型的专用API，可靠性更高。

#### XML标签与预填充

这实际上是两个技巧，配合使用能有效提升输出格式的一致性。XML标签的构建原则是保持一致性（标签名在整个Prompt中保持统一）、嵌套层级（层次结构内容必须嵌套）和语义命名（标签名要能表达内容含义）。预填充是在Prompt结尾添加输出格式的开头部分，作用是可以强制模型跳过前言并直接进入正题，这需要模型API能力上的支持；在角色扮演场景中，预填充可以用于锁定角色发言风格。

### 高级工程技巧

#### 长文本处理技巧

当输入包含多个长文档时，文档的组织方式直接影响输出质量，因而有以下技巧：
1. **文档放在Query之前**：将长文档放在Prompt的开头，query和instructions放在后面，通常能改善响应质量。
2. **使用XML标签结构化多文档**
3. **先引后析**：对于长文档任务，先让模型提取相关引用，再基于引用进行分析。

#### 减少幻觉

幻觉（Hallucination）是LLM的固有缺陷，但可以通过工程手段降低：
1. **显式承认不确定性**：如要求模型在信息不足时直接报告缺少必要信息。
2. **引用验证**：对于涉及长文档的任务，先提取逐字引用，再基于引用分析。
3. **N次最佳验证**：用相同Prompt多次调用模型并比较输出，不一致的输出可能表明存在幻觉。
4. **迭代改进**：将模型输出作为下一轮Prompt的输入，要求验证或扩展先前的陈述。

#### 提高输出一致性

1. **明确输出格式**：使用JSON Schema或XML Schema精确定义输出结构。
2. **预填响应**：通过预填充强制特定格式。
3. **知识库检索一致**：对于需要一致上下文的场景，使用检索将响应建立在固定信息集上

#### 链式提示设计

链式提示设计（Prompt Chaining）将复杂任务分解为多个子任务，每个子任务有独立的Prompt。它有以下设计原则：
1. 识别子任务：将任务分解为连续的步骤。
2. XML交接：使用XML标签在提示之间传递输出。
3. 单一目标：每个子任务只有一个明确的输出目标。
4. 迭代优化：根据执行效果调整单个步骤。

### 安全防护

Prompt注入（Prompt Injection）是指攻击者通过构造外部输入，试图覆盖或篡改Agent的系统指令。

#### 三层防护体系

1. **执行层（权限最小化与沙箱隔离）**：Agent代码执行环境与宿主机物理隔离；API Key、数据库权限严格受限；危险操作必需额外授权。
2. **认知层（Prompt隔离与边界划分）**：利用API原生的Role划分区分System Prompt和User Input；使用分隔符将不可信数据包裹，以阻止攻击指令跨区覆盖。
3. **决策层——人机协同**：对于高危操作，执行前触发中断，推送审批请求给管理员。

#### 越狱与提示词注入的缓解

1. **无害性筛选**：对用户输入进行预筛选，检测涉及有害、非法或露骨活动的内容。
2. **输入验证**：过滤已知越狱模式。
3. **链式保障**：分层策略组合使用，构建防御纵深。

## Context Engineering

**Context Engineering**（上下文工程）指的是从大量可用信息中，筛选出最相关的内容，放进有限的上下文窗口。一个完整的上下文窗口通常包括系统提示词（角色定义、任务描述、输出格式规范）、工具上下文（可用工具定义、函数签名、调用结果）、记忆上下文（短期记忆和长期记忆）和外部知识（RAG 检索结果、数据库查询）。上下文工程和提示词工程的区别在于：提示词工程聚焦于指令本身的撰写和组织编排，核心问题是“怎么措辞、怎么排列”（怎么写好提示词）；上下文工程是构建一套动态系统，核心问题是“什么信息以什么格式在什么时机填入上下文”（怎么帮LLM用好提示词）。

### 上下文工程的主要内容

上下文工程本质上是为LLM构建一个高信噪比的信息输入环境，它直接决定了Agent的智商上限、任务连贯性以及运行成本，是决定Agent能力的关键要素。狭义上，主要聚焦于静态的Prompt结构化设计；广义上，囊括了所有影响LLM当前决策的输入信息管理，涉及以下环节：
1. System Prompt（系统指令）：静态Prompt的结构化编排。核心是把角色设定、目标、约束、执行流、输出格式拆解清楚，让模型在复杂任务里不脱轨。
2. User Prompt：用户输入的业务数据与指令。
3. Memory（记忆系统）：短期记忆（会话级记忆，通过滑动窗口管理）和长期记忆（主要是核心事实提取与向量数据库存储）
4. RAG & Tools（动态增强与挂载）：根据当前对话意图，动态检索外部文档作为背景知识；把各种原子工具或复杂技能的功能描述，以结构化文本的形式挂载到上下文中，让大模型知道当前能调用哪些能力。RAG本质上是上下文工程的一种特定实现模式，因为“检索什么、怎么检索、检索结果怎么填入上下文”这三个问题本身就是上下文工程。
5. Structured Output（结构化输出）：输出格式的定义，直接影响下游消费方的解析和后续Agent链路的衔接，是容易被忽视但实战价值很高的一环。
6. Token Optimization（上下文裁剪与优化）：工程实践中最关键的一环。由于上下文窗口有限，需要引入摘要压缩、无用历史剔除或上下文缓存技术，在保证信息完整度的同时降低Token开销和响应延迟。

其中，System Prompt、User Prompt和Structured Output已经被提示词工程涵盖，而其它环节可以认为是上下文工程才关注的独有重点。

### 上下文工程的核心技术

上下文工程在技术上包含三大核心板块：
1. 静态规则的结构化编排：通过采用高度结构化的格式编排系统提示词，以防止模型在长文本中迷失（业内通常采用Markdown强制划分[Role] 角色设定、[Objective] 核心目标、[Constraints] 严格约束、[Workflow] 标准执行流 以及 [Output Format] 输出格式）。
2. 动态信息的按需挂载：由于上下文窗口有限和注意力机制，必须实现精准的按需加载，通常有工具检索与懒加载（先通过向量检索得到最相关的几个候选工具定义再挂载，避免工具幻觉并节省Token）和动态记忆与RAG（通过滑动窗口管理短期记忆，利用向量数据库或其它外部存储系统检索长期事实，并将外部执行环境的反馈进行摘要脱水后实时回传）两种手段。
3. Token预算与降级折叠机制：复杂工程中的核心挑战。为防止长任务的上下文窗口溢出，通常采用低优先级折叠（如将早期的详细对话历史压缩为AI摘要）、中优先级精简（对RAG检索到的背景资料进行二次裁切，仅保留核心段落）、高优先级绝对保护（不准丢失系统约束与当前核心工具描述，以确保Agent的逻辑一致性）的优先级剔除策略。另外，可以配合上下文缓存技术在大规模并发请求中进一步降低首字延迟和推理成本。

### 上下文失效问题

即使在使用超长上下文模型时，模型生成效果也不是上下文越长越好。真实情况是上下文存在边际效益递减，甚至可能负向增长，其根本原因是Transformer架构的注意力机制，主要包括以下方面：
1. 上下文腐化（Context Rot）现象：随着上下文Token总量增大，模型整体的信息回忆能力随之下降。
2. 中间丢失（Lost in the Middle）问题：模型对位于上下文中间位置的信息记忆力显著低于开头和结尾，呈U型分布。
3. 训练数据集模式：模型的Attention模式是在短序列数据上训练出来的，互联网文本的平均长度远低于现在的上下文窗口。这意味着模型处理长依赖关系时没有足够的学习经验，位置编码的外推能力（推理时能够正确处理长度大于训练数据集中序列的上下文的能力）也有限。虽然有位置编码插值技术来缓解长序列外推问题，但精度损失是结构性的，不会完全消失。

因此，上下文必须被当作有限资源来管理，不是塞越满越好；找到“高信噪比”的平衡点，是上下文工程最核心的手艺。

### 上下文利用率问题

当传给模型的上下文长度超过上下文窗口的40%时，Agent输出质量开始明显下降，根据这个阈值可以把Agent的输出质量划分为两个区间：
1. Smart Zone（0～40%）：推理聚焦、工具调用准确、代码质量高
2. Dumb Zone（>40%）：幻觉增多、兜圈子、格式混乱、低质量代码

另外，Anthropic在实践中遭遇了“上下文焦虑问题”：Sonnet4.5在上下文快填满时会变得犹豫，哪怕任务还没做完也倾向于提前收工；光靠压缩上下文并不足以解决这个问题，他们的最终做法是直接清空上下文窗口，但通过结构化的交接文档把关键状态留下来。工程优化的目标不是给Agent塞更多信息，而是让它在任何时候都运行在干净、相关的上下文里。

### 有效上下文的构建原则

要构建有效的上下文，需要考虑以下方面：
1. “恰到好处”的System Prompt：避免过度设计和过度抽象，保证足够具体以引导行为，同时足够抽象以提供通用启发；建议先用最小化Prompt验证，再基于失败用例逐条补充清晰指令，而不是一开始就试图穷举所有规则。
2. 防止工具描述误导Agent：明确描述什么时候该调用和什么时候不该调用，关键在于边界清晰而非面面俱到；一个工具只做一件事，参数描述要包含格式示例。
3. 合理选择Few-shot示例：预防模型过度拟合表层模式进而忽略底层逻辑，可以选3～5个多样化的典型实例，示例的多样性和典型性比数量更重要，每个示例要能代表一类典型场景的解决模式，而非覆盖所有边缘情况。

### 上下文加载策略

传统AI应用的做法是预检索，即在调用LLM之前通过向量检索出最相关的上下文，然后全部塞进Prompt；这套机制的问题是无法涵盖Agent在执行过程中动态发现的新线索，JIT（Just-in-Time，按需加载）策略因此兴起。

JIT的核心思想是Agent运行时不预装所有可能相关的信息，只维护轻量级的引用句柄，在真正需要时才通过工具动态拉取数据。在Claude Code中采用的渐进式披露策略是一种典型实践：Agent通过层层探索逐步构建对信息的理解，而不是一次性获取全部上下文；每一次交互都揭示新的上下文，进而引导下一步决策。

但是，JIT也有明显的代价：运行时探索比预检索更慢，还需要提供足够好的导航工具防止Agent在信息海洋里迷路，它对工具集和导航策略的设计要求比预检索更高。最优解往往是混合策略：对确定性高的静态知识预检索，对动态发现的信息按需拉取；动态内容占比高、探索空间大的场景适合按需加载为主，动态内容少、上下文稳定的场景适合预检索为主结合少量动态加载。

### 上下文代谢机制

在长时任务中，单纯的上下文管理不够用，需要引入以下机制抵抗上下文腐化：
1. Compaction（压缩）：当上下文窗口快满时，把历史内容交给LLM总结，然后用摘要创建一个新的上下文窗口继续工作；难点在于筛选需要保留的内容，保留太多则压缩无效，保留太少则关键上下文丢失；工程上用复杂Agent轨迹数据反复调优压缩Prompt，即先最大化召回以防漏掉重要信息，再逐步精简冗余内容，对于已被消化使用的工具调用结果可以直接清理。
2. Structured Note-taking（结构化笔记）：让Agent把关键进展以结构化格式写入外部文件，后续基于新上下文重新读取。
3. Sub-agent Architectures（多Agent架构）：让多个Agent共同工作，采用一个主Agent负责任务编排和结果汇总，而让多个专业化的子Agent处理专门任务；这种设计实现了关注点分离，每个子Agent可以探索大量上下文，而主Agent保持了干净的上下文专注于分析和决策，相比单Agent架构在复杂任务上实现了显著的质量提升。

### 上下文工程落地

#### 工具链

1. 编排框架：负责Agent的控制流、状态管理和循环调度，主要有LangChain、LangGraph、agentScope、Qwen-Agent等。
2. 数据框架：专注RAG场景下的数据摄取、索引和检索优化，LlamaIndex是业内的事实标准。
3. 向量数据库：负责向量（Embedding）的存储和语义搜索，主要有Pinecone、Weaviate、Chroma、Qdrant、Milvus、PgVector、Elasticsearch、LanceDB。
4. 通信协议：解决工具标准化接入宿主问题的MCP协议和解决Agent间协作通信的A2A协议。
5. 记忆系统：在向量数据库等存储系统基础上封装记忆完整生命周期管理的平台，主要有Memo、LETTA、ZEP等。

#### 核心原则

1. 上下文是系统输出，不是静态配置：每次LLM调用前都要动态组装上下文，组装逻辑是工程的核心。
2. 高信噪比优于高信息量：找到让模型做出正确决策所需的最小高密度信息集。
3. 上下文需要代谢机制：即长期记忆的演化机制，主要用于长任务的上下文持久化管理。
4. 从最简方案开始，逐步增加复杂度：过度工程化和上下文不足一样危险。

## Harness Engineering

**Harness Engineering**（驾驭工程，下面简称Harness）指围绕LLM搭建的一整套执行与约束层，它就是Agent中模型之外的一切（核心公式：Agent = Model + Harness），包括系统提示词、工具调用、文件系统、沙箱环境、编排逻辑、钩子中间件、反馈回路、约束机制等。Harness与上下文工程、提示词工程是嵌套关系：上下文工程包含了提示词工程，而驾驭工程又包含了上下文工程。这三层分别解决不同的问题：
1. 提示词工程解决表达问题，即怎么写好指令，关注点在塑造局部概率空间，让模型听懂意图。
2. 上下文工程解决信息问题，即给Agent看什么，关注点在确保模型在合适的时机拿到正确且必要的事实信息。
3. Harness解决执行问题，即整个系统怎么防崩、量化、持续运转等，关注点在长链路任务中的持续正确、偏差纠正、故障恢复。

模型决定了系统的上限，Harness决定了系统的底线。Harness Engineering做的事情就是承认模型有边界，然后把边界之外的需求一个个工程化地补上。

### Harness的组成

模型做不到的，就是Harness要补的：
1. 记忆系统：模型无法记住多轮对话历史，通过维护对话历史、每次请求时拼进上下文为Agent补齐。
2. 通用执行环境：模型无法执行代码、跑命令，通过提供Bash + 代码执行环境为Agent补齐。
3. 外部知识获取：模型无法获取实时信息，通过Web Search、MCP 工具为Agent补齐。
4. 文件系统：模型无法操作文件和环境，通过文件系统抽象 + Git版本控制为Agent补齐。
5. 验证闭环：模型无法知道自己做对了没有，通过沙箱环境 + 测试工具 + 浏览器自动化为Agent补齐。
6. 上下文管理：模型无法在长任务中保持连贯，通过上下文压缩、记忆文件、进度追踪为Agent补齐。

其中，上下文管理是上下文工程的核心关注点，而记忆系统和外部知识获取也在上下文工程中有部分涉及，其它部分则是Harness独有的。在LangChain中，Harness的核心组件被拆解为五个子系统，包括文件系统（持久化）、Bash 执行（通用工具）、沙箱环境（安全隔离）、记忆机制（跨会话积累）、上下文压缩（对抗衰减）。

### Harness的六层体系架构

1. 信息边界层L1（该关注什么）：定义角色与目标，裁剪无关信息，结构化组织任务状态；解决Agent该知道什么、不该知道什么的问题，
2. 工具系统层L2（用什么干活）：工具的选拔、调用时机、结果的提炼与反馈；解决Agent怎么跟外部世界交互的问题。
3. 执行编排层L3（按什么步骤做事）：让模型像人一样走完“理解目标→判断信息→分析→生成→检查”的完整轨道；解决多步骤任务怎么串起来的问题。
4. 记忆与状态层L4（怎么记住做过的事）：独立管理当前任务状态、中间产物和长期记忆，防止系统混乱；解决长任务中间结果怎么管的问题。
5. 评估与观测层L5（怎么检验做对了没有）：建立独立于生成过程的验证机制，让Agent具备“自知之明”；解决Agent怎么知道自己做对了没有的问题。
6. 约束、校验与恢复层L6（什么事绝不能做、出了事怎么补救）：预设规则拦截错误，失败时提供重试或回滚机制；解决出错了怎么办的问题。

这个架构最大的价值在于从“定义边界”到“兜底恢复”的完整闭环。建议最初从L1和L6入手，因为这两层的投入产出比最高，中间的层次可以随着项目复杂度增长逐步补齐。

### Harness实践指南

对于构建Harness系统，不需要一开始就备齐所有东西，以下按优先级从高到底排列可以执行的工作：
1. P0（不用犹豫，立即可以做）：创建AGENTS.md（只当目录用，详细规则放在子文档中按需加载）并持续维护（Agent每次启动自动加载，犯错就更新，形成反馈循环）、构建自定义Linter+修复指令（错误消息里直接告诉Agent怎么改，纠错的同时在“教”）、把团队知识放进仓库（写在外部文档里的知识对Agent等于不存在）。
2. P1（P0做完之后，可以考虑这些）：分层管理上下文（不要把所有东西塞进一个文件，采用渐进式披露）、建立进度文件和功能列表（JSON格式追踪功能状态，Agent不易乱改结构化数据）、给Agent端到端验证能力（浏览器自动化让Agent能像用户一样验证功能）、控制上下文利用率（尽量不超过40%，增量执行）。
3. P2（有余力再考虑）：Agent专业化分工（每个Agent携带更少无关信息，留在Smart Zone）、定期垃圾回收（确保清理速度跟得上生成速度）、可观测性集成（把“性能优化”从玄学变成可度量的工作）。

### Harness实践阶段

1. 无Harness（Level0）：直接把Prompt给Agent，无结构化约束
2. 基础约束（Level1）：AGENTS.md + 基础 Linter + 手动测试
3. 反馈回路（Level2）：CI/CD 集成 + 自动化测试 + 进度追踪
4. 专业化Agent（Level3）：多 Agent 分工 + 分层上下文 + 持久化记忆
5. 自治循环（Level4）：无人值守并行化 + 自动化熵管理 + 自修复

### Harness未解问题

Harness Engineering是一个快速发展的领域，仍有许多未解的问题：
1. 棕地项目怎么改造：所有公开案例全是绿地项目，零方法论（绿地项目指从零开始的新项目，没有历史包袱；棕地项目指在已有代码库上改造，有历史架构、技术债、遗留逻辑的约束）；环境本身的结构特性（类型系统、模块边界、框架抽象）决定了Harness能做多好。
2. 怎么验证Agent做对了事：大家擅长“约束不做错事”，但“验证做对了事”远未解决；用AI生成的测试来验证AI生成的代码，本质上是“用同一双眼睛检查自己的作业”。
3. AI生成代码的长期可维护性：LLM代码经常重新实现已有功能，长期效果未知。
4. Harness该做厚还是做薄：这是由场景决定的，通用产品追求最小化，特定产品可以高度定制；而且随着模型变强，已有Harness应该定期简化。
5. 单Agent还是多Agent：这是由规模决定的，小项目单Agent够用，大项目几乎必然需要多个专业化Agent组合。


### Harness实践经验总结

#### 团队实践

1. 给Agent一张地图，而不是一本千页手册（渐进式披露原则）。
2. 架构约束不能写在文档里，必须靠工具强制执行（通过自定义Linter加结构测试保证）。
3. 可观测性也是给Agent看的，不只是给人看的（把监控工具接入Agent）。
4. 熵不会自己消失，必须主动对抗（清理垃圾的速度跟上了生成的速度，系统才能可持续地运行）。
5. 写在外部文档里的知识，对Agent来说等于不存在（所有团队知识都要作为版本控制的制品放置在仓库中）。
6. 日志不往控制台打（用grep友好的单行格式全部写进文件，主动控制上下文污染）。
7. 测试不全部跑（随机分配测试集给多个Agent，让Agent集体覆盖全部测试，避免单个Agent花太长时间跑测试）。
8. Agent角色专业化（随着项目成熟，拆分不同Agent承担专门角色）。
9. 受GAN（Generative Adversarial Networks，生成对抗网络）启发的三智能体架构（旨在解决自我评价偏差问题）：
   1. Planner：把1-4句话的产品描述扩展成完整的产品规格，要求“在范围上要大胆”。
   2. Generator：按功能依次做"Sprint"，每个Sprint有明确的完成标准。
   3. Evaluator：通过端到端自动化工具（如Playwright）测试应用，按产品设计深度、功能性、视觉设计、代码质量等维度打分（设计质量和原创性的权重被故意调得比功能性和代码质量更高，因为模型倾向于做出“功能齐全但长相平庸”的东西，权重调整是在引导它往更难的方向使劲）。
10. 上下文重置（解决“上下文焦虑”问题）：
    1. 当一个Agent的上下文接近饱和时，先把当前任务状态、已完成的工作、待办事项结构化地提取出来。
    2. 启动一个全新的“干净”Agent，把结构化的交接文档交给它。
    3. 新Agent从干净的状态继续工作。
11. 按需混合确定性节点与Agent节点，人只在最后审查。

#### 个人实践

1. 坚持一次只跑一个Agent，保持深度参与。
2. 放弃聊天模式：让Agent在能读文件、跑程序、发 HTTP 请求的环境里直接干活。
3. 复现自己的工作：每件事做两次，一次自己做，一次让Agent做。
4. 下班前启动Agent：每天最后30分钟给Agent布置任务，包括深度调研、模糊探索、Issue分拣等。
5. 外包确定性任务：挑出Agent几乎一定能做好的任务在后台运行，并关掉桌面通知以避免上下文切换。
6. 工程化Harness：每当Agent犯错，就工程化一个解决方案让它永远不再犯同样的错（反馈式更新AGENTS.md，每行规则对应一个过去的失败案例）。
7. 始终有Agent在跑：目标是10~20%的工作时间有后台Agent运行。

## Workflow

**Workflow**（工作流）在这里特指AI场景下的任务执行编排：用一种可分支、可循环、可观测的执行路径组织信息检索、工具调用、输出结构化结果、质量检查、失败重试等流程。它是一套能动态决策、自动修正、可控收敛的执行机制，主要是为了实现AI系统稳定交付结果的需求，旨在解决以下核心问题：
1. 下一步并不唯一，需要根据当前结果动态决策路径。
2. 当结果不理想时，系统需要自动修正，而不是直接失败。
3. 中间状态必须被记录，否则难以调试、追踪与恢复。

传统工作流的核心假设是：给定相同输入，同一节点的执行结果是确定的；而AI工作流不符合这一特点。两者的关键差异在于：传统工作流的路径选择逻辑是预定义（在设计时确定）的，AI工作流的路径选择依赖于运行时生成内容的质量评估，且同一节点可能因输出不确定性而需要反复执行。

### AI工作流的核心概念

Workflow是目标与过程，Graph是结构与载体，Loop是Graph上的控制模式。三者是同一件事的三个观察角度：Workflow关注任务目标；Graph关注结构组织；Loop关注回溯控制。

#### Graph

**Graph**（图）是工作流的结构表达，包含三种基础元素：Node、Edge和State。

**Node**（节点）表示一个执行单元，主要有三大功能，包括读取状态（State）、执行业务逻辑并加工状态、将加工好的状态放回。

**Edge**（边）是图中的控制流抽象，用于描述节点之间的执行路径及其触发条件，决定流程在运行时如何在不同节点之间进行调度与跳转。常见的边有以下类型：
1. 顺序边（Sequential Edge）：按固定顺序执行节点，不依赖条件或状态判断
2. 条件边（Conditional Edge）：在设计时定义有限的候选路径，根据运行时状态选择其一；候选目标节点在设计时确定，运行时只做选择。
3. 动态路由（Dynamic Routing）：目标节点不在设计时完全预定义而由运行时逻辑动态确定，候选集合是开放的。
4. 循环边（Loop Edge）：节点回到自身或前序节点重复执行，通常是由条件边与顺序边结合形成；用于重试、迭代优化或循环推理，直到满足终止条件
5. 终止边（Terminal Edge）：将流程引导至结束状态，不再继续执行后续节点；用于输出最终结果或结束工作流。
6. 并行边（Parallel Edge）：一个节点同时分发到多个后续节点并行执行；用于多任务处理、RAG/工具并发等场景。

**State**（状态）表示在流程执行过程中持续被读写的共享上下文，是节点之间真正传递的“工作记忆”；本质上是一个键值对数据结构，用于在各节点之间传递和修改数据。状态的设计不仅涉及“存什么”，还涉及“怎么更新”，状态有不同的更新语义：
1. 覆盖（Replace）：新值直接替换旧值；适用于单值字段，如分类结果、当前状态。
2. 追加（Append）：新值追加到已有列表；适用于累积型字段，如对话历史。
3. 自定义合并（Custom Reducer）：通过自定义函数决定合并逻辑。

当多个并行节点同时写入同一个使用覆盖语义的字段时，会出现竞态问题，因此设计状态时需要提前规划并行写入字段并选择合适的更新策略。

#### Loop

**Loop**是Graph上提供回溯能力的控制模式，技术上表现为回边（Edge Back）。常见的Loop有固定次数循环和条件驱动循环，其中后者更具有代表性，因为“跑几次”往往不是先验确定的，而是由内容质量、工具执行结果、外部反馈共同决定的；但在实际开发中两者必须同时使用，因为LLM的不确定性可能会导致条件驱动循环无法终止，此时需要结合固定次数循环模式对内容进行降级兜底处理。

在实际工程中，通常采用嵌套循环：外层循环负责“质量迭代”（生成 → 审核 → 修改）；内层循环负责“工具重试”（某个节点内部调用外部API失败后进行指数退避重试）；两层循环的作用域、终止条件、计数器是独立的，设计时需要分别明确独立的退出条件和安全边界。

如果没有合理的约束，Loop很容易陷入死循环，因此一个可靠的Loop一定包含三件事：
1. 继续条件：为什么还要再来一轮。
2. 退出条件：什么时候已经足够好，可以结束。
3. 安全边界：最大轮次、超时、预算、熔断条件。

### AI工作流的设计策略

抽象能力是工作流设计的分水岭，好的工作流关键看Node、Edge、State的抽象能否经得起复用与扩展，和步骤多少关系不大。成熟的方式是把流程抽象到更稳定的结构层：
1. Node抽象职责边界：在这个节点中产出的结果该是什么样子的，必须出现哪些信息，而不是抽象"这一次调了哪个API"。
2. Edge抽象流转规则：在什么状态下允许去哪、何时结束；用条件边表达分支与循环，而不是在图外写满 if-else。
3. State抽象推进任务时必须持久记住的信息：工单快照、审核结论、重试次数、错误码等，让路径有据以依。

### AI工作流的注意事项

1. State设计的粒度：太粗会导致所有东西都塞进一个大对象里，谁改了哪个字段不好查；太细会导致字段拆得特别散，每个节点都要拼来拼去容易出错；建议按业务含义分块。
2. 循环终止条件（避免死循环）：要明确最大轮次、评分阈值、超时或成本超限时的兜底策略、连续失败后是否要fallback。
3. 错误处理与Fallback（降级）：工具异常、模型超时、格式校验失败、外部接口限流等“失败路径”要在图上有明确边：重试、降级、转人工、或输出"当前最优 + 错误说明"，而不只靠外围try-catch吞掉。根据错误类型分别采用不同的处理策略（可以参考分布式系统中的成熟方案，如指数退避重试、熔断降级、舱壁隔离、补偿事务等）：
   1. 瞬时错误：采用指数退避重试（如网络超时、API 限流）
   2. LLM可恢复错误：将错误存入State循环回去让LLM调整策略（如工具调用失败、输出格式异常）
   3. 用户可修复错误：暂停执行，等待人工输入后恢复（缺少必要信息、指令不明确）
   4. 意外错误：让其冒泡交给开发者调试（未知异常）。
4. Token消耗与成本控制：Loop会自然放大Token消耗与延迟，设计时要提前思考：
   1. 哪些节点必须调用大模型，哪些可以用代码替代。
   2. 是否可以先粗筛再精修。
   3. 是否需要在达到"足够好"时就提前结束，而不是追求"理论最优"。
5. 节点间数据传递格式：尽早统一节点之间传什么、字段名怎么定义、结构化输出采用什么Schema，以避免图的复杂度增加时调试成本会急剧上升。

### AI工作流的演进方向

工作流就是一种工程建模能力，常见演进方向包括：
1. Agent化：节点从「固定脚本」变成「能自主选工具、拆子目标」的执行单元，但底层仍需要清晰的图与状态边界，否则难以观测与兜底。
2. 多智能体协作：多个角色分工、对话或委托；难点往往在共享 State 的权限与冲突解决。
3. 人机协同：在关键节点插入人工审核、标注或纠偏，把 HITL（human-in-the-loop）当作一等公民写进图与状态机。
4. 更长上下文与记忆：工作流与 RAG、会话记忆结合时，要特别注意 State 里哪些该进向量库、哪些只该留在本轮任务上下文，避免成本和隐私失控。

### AI工作流的安全问题

#### 通用的安全风险

工作流为LLM输出引入了结构和约束，但也带来了新的攻击面，需要重点关注以下三类威胁：
1. 提示注入的级联影响：恶意用户输入可能覆盖系统提示，在工作流中逐节点传播放大；防御方式包括输入过滤、系统提示与用户输入严格分隔、对LLM输出做安全检测后再传递给下游节点。
2. 工具调用的权限边界：遵循最小权限原则，每个节点只能访问其任务所需的工具，高风险操作必需通过人机协同节点确认。
3. 输出内容安全过滤：LLM输出在进入下游系统前必须经过校验，防止注入攻击、隐私泄露和幻觉传播。

#### 特有的安全考量

1. State污染：恶意输入经节点处理可能将路由控制字段写入State，进而影响后续条件边路由，导致跳过审核直达输出；防御手段是对State中的路由控制字段做白名单校验。
2. Loop放大攻击：恶意输入构造使评估节点永远返回低分，导致Loop总是达到最大轮次才退出，大量浪费Token；防御手段是增加Token消耗预算作为独立的安全边界。

## RAG

**RAG**（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索（Information Retrieval, IR）技术与生成式大语言模型（LLM）相结合的框架。其核心思想是在让LLM回答问题或生成文本之前，先从外部知识源中检索出相关的上下文信息，再将这些信息与原始问题一并提供给LLM，从而使其能够产出更准确、更具时效性、更符合特定领域知识的回答。

RAG是解决以下三个核心挑战的有效方案：
1. 知识时效性问题：预训练LLM的知识通常固化在其训练数据截止时间点，模型无法掌握训练后发生的新事件、新政策、新产品文档等信息；RAG通过动态检索获取外部知识并注入给LLM作补充。
2. 私有数据访问：出于数据安全和商业机密的考虑，企业内部的私有数据无法被公开的LLM直接访问；RAG能够安全连接这些私有数据源，在用户提问时仅提取与问题相关的片段信息提供给LLM，使其在不泄露全部数据的前提下基于企业自身知识进行回答。
3. 提升回答准确性与可追溯性：LLM有时会产生"幻觉"，即编造不符合事实的信息；RAG通过提供明确、有据可查的参考文本，引导LLM尽量基于检索证据回答，从而降低幻觉概率（但RAG不能彻底消除幻觉，生产中还需结合引用校验、答案评估、拒答机制和人工反馈闭环等机制使用）。

RAG与传统搜索引擎在以下方面存在本质区别：
1. 检索机制：传统搜索主要基于倒排索引、关键词匹配和相关性排序；RAG更强调"检索结果进入LLM上下文后参与答案生成"，但不限制检索方式。
2. 处理逻辑：传统搜索本质是相关性排序器，将候选文档按相关性得分排序后直接呈现给用户，每个结果相对独立，不进行跨文档的信息融合；RAG本质是信息综合器，它会将检索到的多个知识碎片（Chunks）喂给LLM，由模型进行逻辑归纳和跨文档的信息整合。
3. 结果交付：传统搜索提供候选文档列表（线索），需要用户二次阅读过滤；RAG提供的是答案，能直接回答复杂指令，并通过引文标注（Citations）兼顾信息的来源可追溯性。
4. 时效性与数据范围：传统搜索更依赖大规模爬虫和全网索引；RAG则常用于私有知识库或垂直领域，能低成本地让LLM获得实时或特定领域的知识补充。

### RAG的工作原理

RAG的工程链路通常分为两个阶段：离线索引阶段与在线的检索增强生成阶段。索引阶段负责把原始文档处理成可检索的数据结构；在线阶段则在用户提问时完成查询理解、检索召回、上下文构建和答案生成。

#### 索引阶段

在索引阶段，文档会进行预处理，以便在检索阶段实现高效搜索，通常包括以下步骤：
1. 输入文档：文档是需要被处理的内容来源，可能是文本文件、PDF、网页、数据库记录等。
2. 清理文档：对文档进行去噪处理，移除无用内容（如HTML标签、特殊字符）。
3. 增强文档：利用附加数据和元数据（如时间戳、分类标签）为文档片段提供更多上下文信息。
4. 文档拆分（Chunking）：通过文本分割器将文档拆分为较小的文本片段（Segments），需要兼顾语义完整性、Embedding模型输入长度、生成模型上下文窗口和召回粒度（Chunk过大易引入噪声，过小又可能丢失上下文，拆分策略会直接影响召回质量）。
5. 向量化表示（Embedding Generation）：通过嵌入模型将文本片段映射为语义向量表示（Document Embedding，即高维稠密向量）。
6. 存储到向量索引系统：将生成的嵌入向量、原始内容及其对应的元数据存入向量存储或向量索引系统中（如 Milvus、pgvector、Elasticsearch/OpenSearch向量检索等）。

索引过程通常是离线完成的，对于动态需求也可以在线完成。

#### 检索阶段

检索通常是在线进行的。当用户提交一个问题时，系统会使用已索引的文档来回答问题。该阶段通常包括以下步骤：
1. 接收请求：接收用户的自然语言查询（Query），某些场景可以对原始查询进行改写或扩充以提高后续检索的覆盖率。
2. 查询向量化：使用嵌入模型将用户查询转换为语义向量表示（Query Embedding），以捕捉查询的语义信息。
3. 信息检索（Retrieval）：在向量存储中通过语义相似性搜索找到与查询向量最相关的文档片段（Relevant Segments）。
4. 上下文增强（Augmentation）：将检索片段、原始问题、系统指令和引用要求组织成Prompt输入给LLM，引导模型基于检索证据回答问题。
5. 输出生成（Generation）：向用户输出自然语言回复，并附带相关的参考资料链接。
6. 结果反馈（可选）：允许用户提供反馈，以便于调整提示词或检索方式来优化生成效果；在某些实现中，通过支持多轮交互进一步完善回答。

#### Embedding

**Embedding**指文本的向量化过程，可以理解为“把文本变成一串数字”：它会把文本映射到一个高维稠密向量空间里，让语义相近的文本在向量空间中距离更近。Embedding模型不是“实时理解世界”的模型，它只负责把文本映射到向量空间，主要能力是语义匹配；如果遇到非常新的术语、梗、产品名或领域缩写，需要通过业务语料评测确认召回效果。

文本变成向量之后，检索系统需要判断“哪个向量和查询最接近”，有以下三种常见度量标准：
1. 余弦相似度（Cosine Similarity）：看两个向量方向是否一致；特点是对向量长度不敏感，RAG场景最常用。
2. 内积（Inner Product / Dot Product）：看两个向量对应维度乘积之和；如果向量已经L2归一化，内积和余弦相似度在排序结果上通常等价。
3. 欧式距离（L2 Distance）：看两个点在空间中的绝对距离；对向量幅度更敏感，适合模型或索引明确按L2训练/优化的场景。

RAG关注的是语义方向是否接近，而不是向量长度本身，所以余弦相似度通常更适合文本语义检索。

### RAG的核心优势与局限性

RAG的核心优势和局限性可以从知识管理、工程落地和性能指标三个维度来分析。

#### 核心优势

1. 知识时效性与低维护成本：只需要更新知识库、索引和元数据，模型就能获取最新信息，非常适合处理频繁变动的数据；相比微调或重新训练模型，知识更新成本和周期都更可控。
2. 显著降低幻觉并提供引文追溯：将模型从“基于参数化记忆生成”转变为“基于检索证据生成”。每个回答都有明确的信息来源，提供了关键的可解释性和可验证性，这对高准确性场景尤为关键。
3. 数据安全与细粒度权限控制：可以在检索层实现精准的多租户隔离和访问控制，确保用户只能检索其权限范围内的数据；相比把敏感数据放入微调训练流程，更容易做数据隔离和合规治理。
4. 领域适应性强：无需针对特定领域重新训练模型，只需构建领域知识库即可快速适配垂直场景。

#### 局限性与工程挑战

1. 严重的检索依赖性：遵循 GIGO（Garbage In, Garbage Out）原则，输入信息的质量好坏对最终生成答案的效果影响极大。
2. 上下文窗口与推理噪声：注入过多无关片段会造成注意力稀释，干扰模型的逻辑推理，并带来不必要的Token开销。
3. 首字延迟（TTFT）增加：完整链路包含"查询改写 -> 向量化 -> 相似度检索 -> 重排序(Rerank) -> 上下文构建 -> LLM 生成"等多个环节，每个环节都增加延迟。
4. 工程复杂度高：需要维护向量数据库、处理文档更新的增量索引、优化检索策略等，相比纯LLM应用复杂度大幅提升。
5. 长文本Token成本：单次请求携带大量上下文会导致推理成本显著高于普通对话。

### RAG的常见用途

RAG最适合用在"答案依赖外部资料、且资料会变化/很长"的场景：先从知识库检索相关内容，再让大模型基于检索结果生成回答，从而减少胡编、提升可追溯性。常见用途包括：
1. 客服机器人：基于产品知识库做问答、排障、流程引导。
2. 研发/运维Copilot：检索代码库、接口文档、告警手册，辅助定位问题与生成修复建议。
3. 医疗助手：检索指南/药品说明/院内规范后生成辅助建议（不做最终诊断）。
4. 法律咨询：基于法规条文/案例/合同模板检索，生成条款解释与风险提示。
5. 教育辅导：从教材/讲义/题库检索知识点，生成讲解与例题步骤。
6. 企业内部助手：连接制度、SOP、会议纪要、技术文档做检索/总结/对比。
7. 其他：投研/合规/审计（报告/披露/内控）；销售/方案支持（产品手册/标书模板、生成方案并标注出处）。

### GraphRAG

**GraphRAG**（Graph-based Retrieval-Augmented Generation，基于图的检索增强生成）是在传统向量检索之外引入知识图谱，把文档中的实体、关系和结构化上下文显式建模为图结构，在检索时除了召回相似片段外，还会沿着图关系收集证据链，再交给大模型生成答案。

对于复杂知识问答，答案往往不在一个片段里，而在片段之间的关系里。传统向量RAG有以下局限性：
1. Chunk是信息孤岛：切块是向量RAG的必要工程手段，但它天然会打断上下文；向量检索只能判断“哪个文本块和问题最像”，却不知道这些文本块在业务上属于同一个对象，就是因为语义相似不等于关系完整。
2. 向量相似度不擅长多跳推理：向量RAG可能召回相关片段，但不具备沿着关系链路扩展证据的能力。
3. 全局性问题很难靠Top-k片段回答：对于需要对整个语料做聚合、归纳和主题分析的问题，Top-k检索只能看到局部窗口，容易出现两种失效（召回片段太少，看不到整体模式；召回片段太多，Token成本和噪声一起爆炸）；调高Top-k + 重排优化 + 查询改写的方案短期内能缓解此问题，但仍然无法避免用片段相似度解决结构推理的底层问题。

GraphRAG解决的就是上述问题。GraphRAG和传统向量RAG的本质区别在于检索对象变了：传统向量RAG检索的是文本块（Chunk），擅长判断“这段话和我的问题像不像”；GraphRAG检索的是"知识关系网"里的节点、边、路径、社区摘要，再结合原始文本证据回答问题，更擅长理解“这些对象之间到底怎么连起来”。

普通向量RAG主要检索文本Chunk，适合局部事实问答；GraphRAG会把文档中的实体、关系和主题结构显式建模成知识图谱，查询时不仅可以按语义找片段，还可以沿着图关系做多跳检索，或者利用社区摘要回答全局问题。它的优势是关系推理、全局归纳和可解释性更好，代价是构建成本、实体消歧、关系抽取、增量更新和权限控制都更复杂。对于简单文档问答或数据量小、关系不复杂的场景，向量RAG + 混合检索 + rerank往往更划算；GraphRAG应用在明确因多跳关系、跨文档归纳和结构化约束等原因导致向量RAG表现不佳的场景。

#### 核心概念

下面拆解GraphRAG中的关键词。

**知识图谱**（Knowledge Graph）把知识变成可遍历的关系网，本质上是一种用"节点 + 边 + 属性"表达知识的结构，其中：节点（Node）表示实体或概念；边（Edge）表示实体之间的关系；属性（Property）挂在节点或边上作为补充信息。

**实体**（Entity）是图谱里的核心节点，也是GraphRAG的最小业务对象。实体抽取得好不好直接决定GraphRAG的上限，太粗则图谱没有细节，太碎则到处都是重复节点和噪声。做实体抽取很像做领域建模，工程实践中有以下要点：
1. 用JSON Schema强约束抽取格式：避免自由文本解析，降低后处理成本。
2. Few-shot示例覆盖正反例和边界例：告诉LLM什么不该抽。
3. 设置最大实体数上限：防止LLM在长文本中过度抽取。
4. 每个实体强制要求保存它在原始文档中出现的具体位置或文本片段范围的引用：用于溯源和人工校验。

**关系**（Relationship）是GraphRAG真正比向量RAG多出来的东西，也是它的灵魂。GraphRAG会尝试把关系显式化：有了关系，检索就可以沿着路径扩展，这也是GraphRAG能处理多跳问题的关键。

**社区发现**（Community Detection）指从一堆节点中找主题群，是图算法中的常见任务，目标是把图中连接更紧密的一组节点聚成一个社区。在GraphRAG中，社区可以理解为“语料中自然形成的主题群”。常见做法是先从文本中抽取实体、关系和关键声明，再用Leiden或Louvain等社区发现算法构建层级社区，最后为每个社区生成摘要。这样查询全局问题时不必把所有原始文档都塞给LLM，而是先看更高层的社区摘要。

在GraphRAG中，用不同的检索方式处理不同的问题：
1. 局部检索（Local Search）：适合回答围绕具体实体的问题，典型流程是先定位实体，再沿着实体邻居、关系路径、相关原文片段扩展上下文，它在目标上和传统RAG是一致的。
2. 全局检索（DRIFT Search）：适合回答跨语料的整体性问题，典型流程是先利用社区摘要或主题摘要做聚合，再让LLM进行归纳和排序。
3. 漂移检索（Global Search）：局部检索的增强版，从实体邻居扩展时引入社区摘要作为附加上下文，平衡精确性和全局视野；当问题既有实体焦点又需要跨社区关联时，DRIFT比纯局部检索更有优势。

#### 工作原理

GraphRAG的工程链路同样分为两个阶段：构建阶段（从文档到图谱）与查询阶段（从问题到答案）。

构建阶段的核心工作是从文档构建图谱，这是GraphRAG落地成本高的根本原因：它把"检索前处理"从简单的文本切块升级成了一个知识建模和数据治理工程。通常包括以下步骤：
1. 文档解析：从 PDF、网页、Markdown、数据库记录中提取文本；关键风险是OCR错误、表格丢失结构、文档版本混乱等。
2. 文本切分：把长文档切成TextUnit或Chunk；切分太碎会丢关系，切分太大会增加后续抽取成本。
3. 实体抽取：识别文档里的系统、人、组织、概念、事件等实体；同名实体/别名/缩写的歧义和噪声实体是主要难点。
4. 关系抽取：识别实体之间的依赖、包含、影响、因果等关系；可能出现关系方向错、关系类型泛化、置信度不足等常见问题。
5. 图谱归一化：合并重复实体，补充属性和来源信息；实体消歧成本高，需要人工规则和评测。
6. 社区发现：找出连接密集的主题群；图太稀或太脏时社区质量会下降。
7. 摘要生成：为社区、实体、关系生成摘要；依赖LLM调用，可能丢约束或引入幻觉。
8. 索引入库：写入图数据库、向量库、全文索引；需要考虑增量更新和权限过滤策略的复杂性。

查询阶段的关键步骤是查询路由，即判断问题类型并选取合适的检索方式：
1. 局部事实：适合向量检索或局部图检索。
2. 实体关系：适合局部图检索。
3. 多跳推理：适合图遍历 + 向量补证据。
4. 全局归纳：适合全局检索（会先走社区摘要）。
5. 精确过滤：适合图查询或结构化查询。

#### 适用场景

GraphRAG最适合"关系比文本相似度更重要"的场景。它不是向量RAG的默认升级包，而是一套更重的数据治理和检索架构。判断要不要上GraphRAG的核心是看问题的失败原因是否集中在关系、路径、全局主题和跨文档归纳上。

适合GraphRAG的典型场景如下：
1. 企业知识库的复杂问答：问题需要跨部门、跨制度、跨项目复盘串联信息。
2. IT架构和故障影响分析：服务、接口、数据库、消息队列、负责人、告警、事故之间天然有依赖关系。
3. 金融、风控、合规、供应链：这些领域更关心对象之间的关系，而不是文本片段是否相似。
4. 跨文档主题归纳：分析访谈记录、调研报告、客服工单、事故复盘的整体模式时，社区摘要可以先把语料聚成主题群，再让LLM做全局归纳。

不适用GraphRAG的情况如下：
1. 数据量小、问题简单：知识库只有几十篇文档且问题基本都是"某个规则是什么"时，向量RAG + 混合检索 + rerank往往更划算。
2. 文档质量太差：源文档主语缺失、版本混乱、术语不统一、表格解析错误严重等，会导致抽取出的图谱很脏，进而导致"整张关系网方向错了"。
3. 实时性要求极高：实体关系抽取、社区发现、摘要生成都会增加更新成本，数据必须秒级可见时需谨慎评估增量图更新和摘要刷新成本。
4. 团队缺少图建模和评测能力：没有人负责处理"哪些实体值得建模、关系类型怎么设计、实体如何消歧、图谱错误怎么评测、权限过滤放在哪里"等问题时，GraphRAG很容易变成昂贵且不可控的黑盒。

如果失败原因只是“没搜到那段话”，先优化检索；如果失败原因是“搜到了很多话，但系统不理解它们之间的关系”，再考虑 GraphRAG。

#### 落地难点

GraphRAG最容易被低估的地方是"把一堆文本变成可用关系网"后还要长期维护它：实体要抽得准，关系方向不能错，图谱要能更新，权限不能泄露，效果还要能评测。

1. 实体容易抽重、抽错、抽太碎：同一个实体可能有多个名字，需要配术语词典、别名表、规则匹配、人工校验、置信度阈值和评测集来处理实体合并；实体消歧做不好，图谱会变成一堆重复节点，检索路径也会断。
2. 关系方向一错，答案就会系统性跑偏：关系比实体更容易出错，必需给关系加上特定字段用于追溯来源文档、追溯原文位置、记录抽取置信度、控制关系类型、支持增量更新和让LLM升级后做差量重抽与A/B对比；没有来源追溯的图谱不要直接用于高风险问答。
3. 社区摘要成本问题：构建阶段需要大量LLM调用（抽取实体关系、生成实体描述、生成社区摘要、后续版本更新时刷新摘要）。语料很大时，索引成本可能明显高于普通向量RAG；建议先用小语料验证收益，再决定是否引入多层社区摘要和全局检索。
4. 更新一篇文档，可能牵动一片图：GraphRAG更新一篇文档可能影响实体节点、关系边、社区划分、社区摘要、实体摘要、向量索引、权限索引等。每次都全量重建则成本高，做增量更新则工程复杂度高。做GraphRAG不是只维护索引，而是在维护一个会持续变化的知识结构，这也是它比普通RAG更像数据工程的地方。
5. 权限过滤不能只看文档级别：企业知识库绕不开权限，GraphRAG中除了在检索前或检索时做元数据过滤外，还要额外考虑关系（边）联通导致的实体（节点）访问权限传递性问题。尤其是社区摘要，如果其中部分文档对用户不可见，需要防止它变成隐形泄漏点，有如下应对策略：
   1. 社区摘要按权限分组生成：每个权限组独立生成摘要，查询时只返回用户有权限的社区摘要。
   2. 摘要溯源字段保留所有源文档ID：查询时校验用户权限与源文档ID的交集，过滤无权限的证据。
   3. 高敏感语料不参与社区聚合：单独走局部检索通道，避免跨文档泄露。

### LLM Wiki

**LLM Wiki**（大模型百科）是Andrej Karpathy（OpenAI联创，Vibe Coding概念提出者）于2026年4月提出的一种基于LLM构建个人知识库的方案。其核心思想是利用LLM的文本理解和组织能力，将分散的源文档（文章、论文、代码仓库、数据集、图片等）通过LLM增量"编译"成一个结构化的Markdown Wiki文件集合，再让Agent基于Wiki进行问答、分析和知识管理。与传统RAG方案不同，它不依赖特定的数据库系统与检索机制，而是把"知识存储与组织"的工作交给LLM本身完成。

#### 核心流程

1. 摄取（Ingest）：将各类源文档（论文、网页文章、代码仓库、数据集、图片等）收集并存入`raw/`目录。
2. 编译（Compile）：让LLM逐步读取`raw/`目录中的原始内容，并构建一个结构化的Wiki，包含：
   1. 索引文件：包含所有文档的简要摘要，把这些文件作为查询的入口。
   2. 概念条目：按主题组织，附带反向链接和交叉引用。
   3. 衍生输出：幻灯片、图标、归档查询答案等。
   4. 链接图：LLM自动维护概念之间的链接图，用于寻找新条目候选的连接。
3. 查询与增强（Query and Enhance）
   1. Obsidian：用于浏览Wiki与可视化。
   2. 问答Agent：用于回答跨条目的复杂调研问题，用Markdown、幻灯片、图表等形式呈现答案。
   3. 搜索引擎：基于Vibe Coding搜索Wiki，可以通过网页界面或大模型命令行工具使用。
   4. 归档结果：将查询的输出归档回Wiki中，以积累每次探索成果。
4. 清理与维护（Lint and Maintain）：
   1. 扫描数据不一致。
   2. 通过网络搜索补全缺失信息。
   3. 寻找可能成为新文章的概念之间的联系。
   4. 提出更深入的问题用作探讨。
   5. 回到第二步（编译）：Wiki不断增长和改进。

#### 主要特点

1. 不需要向量数据库：在个人知识库规模下，索引文件 + LLM上下文窗口足够检索。
2. 探索总是积累：每个查询、图表和答案都会被归档回维基。
3. LLM负责写作：很少需要手动编辑Wiki，LLM负责编译、链接和维护。
4. 增量编译：新的原始数据被整合进现有的Wiki结构，而不是从头重新处理。
